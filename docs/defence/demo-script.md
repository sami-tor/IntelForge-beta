# IntelForge Defence — Live Demo Script

Total runtime: **~5 minutes**. Rehearse three times before the actual
defence.

---

## 0. Pre-defence checklist (do this 30 minutes before)

```bash
docker ps --filter name=intelforge-postgres   # must be "Up (healthy)"

# Apply both migrations (idempotent — safe to re-run)
docker exec -i intelforge-postgres psql -U intelforge -d intelforge \
  -f - < scripts/intel-automation-migration.sql
docker exec -i intelforge-postgres psql -U intelforge -d intelforge \
  -f - < scripts/intel-automation-v2-migration.sql

# Optional but recommended: backfill 14 days of history so forecasts populate
docker exec -i intelforge-postgres psql -U intelforge -d intelforge \
  -f - < scripts/seed-trend-history.sql

# Boot the app
npm run dev   # (in a separate terminal)

# Bootstrap data
curl -X POST http://localhost:3000/api/cron/intel-sync \
  -H "Authorization: Bearer $env:CRON_SECRET"
curl -X POST http://localhost:3000/api/cron/automation \
  -H "Authorization: Bearer $env:CRON_SECRET"

# Sanity-check the test suite once
npm run defence:test
```

If any of the above fails, **fix it before the defence**, not during.

## 1. Open the Threat Command Center  (00:00 — 00:45)

URL: `http://localhost:3000/intelligence/command-center`

**Talking points:**

- "This is the Threat Command Center. It is the user-facing surface of the
  automation layer I built on top of IntelForge."
- "Top-left you can see the **live SSE indicator**. The page is server-
  rendered for instant load and a tiny client component opens a Server-
  Sent Events stream so the score updates without polling. The endpoint is
  at `/api/intel/automation/stream`."
- Point at the threat score gauge: "This is a composite 0–100 score derived
  from ten cached metrics — KEV count, fresh CVEs, ransomware victims,
  exploits, phishing, malware, dark-web, and feed health. The math is in
  `lib/intel/automation/threat-score.ts:128-176`."

## 2. Walk through one section per minute  (00:45 — 03:30)

### Briefing card

"This is today's executive briefing. It is **deterministic** — built from
real numbers, not an LLM call. So the output is always reproducible and
there is nothing to mock during my demo."

→ click PDF button: "And it exports as a polished PDF in one click."
The PDF download will be a real `%PDF` file generated by PDFKit.

### Action queue preview + open the queue page

URL: `/intelligence/action-queue`

"The pipeline reads my correlation clusters and anomalies and emits
prioritised tasks here. Each one has a category — patch, hunt, block,
review, drill — and a suggested-step playbook. I can mark them in
progress, done, or dismissed. The state machine is documented in
`docs/defence/diagrams/05-state-machine.md`."

### Deep correlation clusters

URL: `/intelligence/clusters`

"This is the deep correlator. It runs three parallel anchor passes —
CVE-anchored, actor-anchored, and ransomware-anchored. Each cluster
bundles up to 11 signal types with per-signal confidence scores. Let me
click on CVE-2023-34362 — you can see it ties together the KEV listing,
a public exploit PoC, a news article, a pastebin victim dump, the linked
actor Cl0p, three named ransomware victims with sectors and countries,
and two dark-web blog posts. That's 11 signals in one cluster, all
chained through actor linkage. The code is in
`lib/intel/automation/correlator-v2.ts`."

### Forecasts + anomalies

Back to command-center, scroll to forecasts:

"These are 7-day forecasts using **Holt's linear exponential smoothing**.
The shaded band is the ±1.96σ confidence interval. The implementation
is at `lib/intel/automation/forecast.ts:69-101` — no ML library, just
the standard textbook formula because with 14 daily points per metric
that's the right complexity."

### Geographic heatmap + sector index

"And finally the geographic and sector views aggregate ransomware
victims, phishing infrastructure and dark-web mentions over a rolling
30-day window. Risk score is rank-derived so the top country always
sits near 100."

## 3. Show the orchestrator in code  (03:30 — 04:00)

Open `lib/intel/automation/orchestrator.ts`

"`runFullAutomation` runs eight stages, each in its own try/catch, and
records every run in `intel_automation_runs`. So if a single stage
fails the others still complete and we know exactly which one
errored."

## What NOT to do during the demo

- Don't open developer tools — keep the browser clean.
- Don't claim more than the implementation actually does. Every claim is
  cited; if you can't cite it, don't say it.
- Don't try to live-edit code. If a question requires code change,
  note it as future work and move on.
- Don't apologise for missing features. The deliverable is what's there.
